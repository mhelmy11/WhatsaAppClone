# MongoDB ID Strategy - ObjectId Only

## Overview

This document explains the decision to use **ObjectId exclusively** for message identifiers, eliminating the dual ID strategy (ObjectId + GUID).

## Why ObjectId Only?

### 1. **Built-in Timestamp**

- ObjectId contains creation timestamp in the first 4 bytes
- Automatic chronological sorting without separate createdAt index
- Range queries by time are more efficient

### 2. **MongoDB Optimized**

- Native MongoDB format - no conversion overhead
- Smaller index size (12 bytes vs 16 bytes for GUID)
- Better performance for lookups and joins

### 3. **Globally Unique**

- Generated from timestamp + machine ID + process ID + counter
- No collision risk across distributed systems
- No need for central ID generation service

### 4. **Simplified Codebase**

- Single source of truth for message identity
- No synchronization between ObjectId and GUID needed
- Cleaner API responses and queries

## Implementation Details

### Message Document

```json
{
  "_id": ObjectId("507f1f77bcf86cd799439011"),  // Primary key only
  "chatId": "UUID",                             // Reference to SQL Server Chat/Group
  "content": "Hello",
  "metadata": {
    "replyToMessageId": ObjectId("..."),        // References another message's _id
    "isForwarded": false
  }
}
```

### C# Model

```csharp
public class MongoMessage
{
    [BsonId]
    [BsonRepresentation(BsonType.ObjectId)]
    public string Id { get; set; }                // ObjectId as string

    public MessageMetadata Metadata { get; set; }
}

public class MessageMetadata
{
    [BsonRepresentation(BsonType.ObjectId)]
    public string? ReplyToMessageId { get; set; }  // References another message's Id

    public bool IsForwarded { get; set; }
}
```

## Collections Using ObjectId References

### Message References (ObjectId)

- `messages._id` - Primary message identifier
- `message_mentions.messageId` - References messages.\_id
- `message_threads.rootMessageId` - References thread root messages.\_id
- `polls.messageId` - References messages.\_id
- `message_search.messageId` - References messages.\_id
- `message_media_index.messageId` - References messages.\_id
- `starred_messages.messageId` - References messages.\_id

### Chat/Group References (GUID from SQL Server)

- `messages.chatId` - References SQL Server Chats or Groups table
- All other collections with `chatId` - References SQL Server

### User References (String from ASP.NET Identity)

- `messages.senderId`, `receiverId` - ASP.NET Identity UserId
- All `userId` fields - ASP.NET Identity UserId

### Status Documents (Separate Feature)

- `statuses._id` - ObjectId (primary identifier, auto-generated by MongoDB)
- **Note**: Use ObjectId returned from insert operation

## Query Examples

### Get Message by ID

```csharp
var message = await _messagesCollection
    .Find(m => m.Id == messageId)
    .FirstOrDefaultAsync();
```

### Get Messages by Time Range (using ObjectId timestamp)

```csharp
var startId = ObjectId.GenerateNewId(startDate);
var endId = ObjectId.GenerateNewId(endDate);

var messages = await _messagesCollection
    .Find(m => m.Id >= startId.ToString() && m.Id <= endId.ToString())
    .SortBy(m => m.Id)
    .ToListAsync();
```

### Get Replied Message

```csharp
var message = await _messagesCollection
    .Find(m => m.Id == messageId)
    .FirstOrDefaultAsync();

if (message?.Metadata?.ReplyToMessageId != null)
{
    var repliedMessage = await _messagesCollection
        .Find(m => m.Id == message.Metadata.ReplyToMessageId)
        .FirstOrDefaultAsync();
}
```

## Benefits Summary

| Aspect                | Dual ID (Old)         | ObjectId Only (New) |
| --------------------- | --------------------- | ------------------- |
| **Storage**           | 28 bytes (12 + 16)    | 12 bytes            |
| **Index Size**        | Larger (2 indexes)    | Smaller (1 index)   |
| **Query Performance** | Good                  | Better              |
| **Code Complexity**   | Higher                | Lower               |
| **Timestamp Info**    | Separate field needed | Built-in            |
| **Sorting**           | Requires createdAt    | Natural by \_id     |

## Migration Notes

### What Changed

1. Removed `messageId` GUID property from `MongoMessage`
2. Changed all message reference fields from `Guid` to `string ObjectId`
3. Updated MongoDB schema JSON to use ObjectId type
4. Removed messageId unique index (only \_id index needed)
5. Simplified metadata (removed clientMessageId, changed forwardedFrom to isForwarded)

### What Stayed

- SQL Server continues to use GUID for Groups, Chats
- ASP.NET Identity continues to use string for UserId
- Status documents use ObjectId \_id as primary identifier (removed redundant statusId)
- ChatId/GroupId references remain GUID (SQL Server PKs)

## Best Practices

1. **Always use \_id** for message identity
2. **Convert ObjectId strings** when passing between clients/servers
3. **Use ObjectId.GenerateNewId()** for time-based queries
4. **Index on \_id is automatic** - no need to create separate index
5. **Store ObjectId as string in C#** with BsonRepresentation attribute

## Conclusion

The ObjectId-only strategy simplifies the architecture, reduces storage requirements, improves query performance, and provides natural chronological ordering. This aligns with MongoDB best practices and WhatsApp-like messaging behavior.
